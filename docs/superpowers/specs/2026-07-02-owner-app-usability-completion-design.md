# 业主端可用性闭环设计

## 目标

本轮将现有 Flutter 业主端从演示原型完善为可顺畅测试的本地 MVP。重点完成“我的”页面、预约回访入口、消息交互和“我的家”项目操作，消除所有可见但无响应的控件。

本轮使用本地持久化数据，不接入真实登录、支付、即时消息或后端 API。架构需允许后续用远程仓库替换本地仓库，而不重写页面。

## 实施策略

采用渐进式本地数据层：保留现有页面和品牌视觉，新增统一的数据模型、仓库接口及本地持久化实现，再逐个打通页面入口。只对本轮涉及的超大页面做必要拆分，不进行无关重构。

## 导航与页面

底部导航保持“首页 / 我的家 / 消息 / 我的”四项不变。

### 我的

“我的”采用个人中心结构：

- 个人资料卡展示头像、称呼、认证状态、城市和项目数量，并支持编辑。
- “我的服务”包含我的预约、在线咨询、我的收藏、平台客服。
- 管理入口包含地址管理、保障与售后、帮助与反馈、设置。
- 每个入口都有独立页面、空状态、主要操作、结果反馈和明确的返回路径。

本轮不提供“发票与合同”入口。未来材料发票只在材料采购订单详情中申请。

### 我的家

打通以下入口：

- 切换项目和编辑项目。
- 项目成员、项目群聊、项目设置和分享。
- 今日提醒的查看与处理，处理结果持久保存。
- 下一步计划、工人详情和全部工人。
- 验收记录、装修档案、资金托管和售后。

资金托管页面只说明流程、保障范围和当前状态，不模拟真实支付。

### 消息

消息中心支持：

- 按关键词搜索消息。
- 按全部、未读和消息类别筛选。
- 全部标记已读和单条标记已读。
- 工人消息进入聊天页面。
- 系统、订单和工单消息进入通知详情页。
- 消息未读数量在页面和导航状态中保持一致。

### 预约与收藏

- 已创建的预约可从“我的预约”再次进入查看。
- 预约列表提供清晰的空状态和继续找师傅入口。
- 师傅可收藏或取消收藏，收藏结果进入“我的收藏”并持久保存。

## 数据架构

新增按领域划分的模型与仓库接口，本地实现负责序列化和持久化。页面通过状态控制器使用仓库，不直接读写存储。

本地数据覆盖：

- 业主资料和地址。
- 项目、成员、提醒、阶段、验收和档案。
- 预约订单。
- 收藏师傅。
- 消息及未读状态。
- 通知、隐私和通用设置。
- 售后申请和意见反馈。

首次启动写入一致的演示数据；之后保留用户修改。演示城市、师傅服务区域、项目日期和订单状态从统一数据源生成，避免成都/深圳或新旧日期冲突。

## 交互规则

- 每个可点击控件必须产生导航、状态变化或明确反馈。
- 表单提交前进行必填项和格式校验。
- 删除、取消等不可逆操作需要确认。
- 成功与失败均提供可理解的页面反馈。
- 列表为空时说明原因，并提供合理的下一步入口。
- 不具备真实能力的功能不伪装成功。
- 页面返回后立即显示最新状态。

## 边界

本轮不实现：

- 真实手机号登录和认证。
- 后端 API、远程数据库和跨设备同步。
- 真实支付、退款和资金托管交易。
- 真正的实时聊天和推送通知。
- 材料采购发票。
- 报价 Web、工长端和管理后台的改造。

## 测试与验收

自动化测试覆盖：

- 四个底部导航页面正常打开。
- “我的”全部入口可以进入并完成主要操作。
- 地址新增、编辑、删除以及重启后的持久化。
- 预约订单能从“我的”找到。
- 消息搜索、筛选、已读和详情跳转。
- 提醒处理、项目切换和验收记录。
- 收藏、反馈和设置状态更新。
- 常见手机宽度下无布局溢出和关键操作遮挡。

完成标准：

- `flutter analyze` 无 error。
- `flutter test` 全部通过。
- 在手机宽度下实际走查首页、预约、消息、我的家和我的。
- 可见界面中不存在空的 `onTap` 或 `onPressed`。
- 未改动本轮边界外的产品。
