# 多师傅候选、上门确认与报价闭环设计

## 1. 目标

在现有业主端、工人端、Spring Boot、MySQL 和 ECS 部署基础上，完成从一条装修需求选择多位候选师傅，到双方确认上门、分别报价、业主独享比价并最终选择一位师傅的真实闭环。

本设计解决以下现状问题：

- 当前一个预约等同于一条装修需求，无法表达同一需求下多位候选师傅。
- 预约接单后缺少双方确认上门时间与到场确认，无法可靠限制“上门后报价”。
- V6 报价只保存粗粒度费用，缺少完整项目、规格、数量、单位、固定平台价和历史版本。
- 工人端后端提交失败会被吞掉，业主“我的家”仍展示本地模拟报价。
- 现有按预约查询报价缺少严格的业主/工人所有权校验。
- 业主确认报价缺少防误触设计，也无法自动结束其他候选预约。

## 2. 已确认的业务规则

- 同一装修需求最多选择 3 位候选师傅。
- 师傅分别上门、分别报价；每位师傅只能看到自己的预约和报价。
- 只有需求所属业主可以看到候选师傅列表和横向报价对比。
- 平台项目单价固定，工人不可修改；工人只能选择项目、规格、数量并填写备注。
- 工人只能在业主确认其已上门后提交报价。
- 业主可确认报价，或填写原因拒绝并要求重报。
- 拒绝后工人创建新版本，历史版本永久保留；待处理报价不能被直接覆盖。
- 业主最终确认一位师傅后，其他候选预约和待处理报价自动结束，但历史记录不删除。
- 业主或师傅均可在确认上门前取消，必须记录操作者和原因。
- 业主确认已上门后不能普通取消，只能进入后续“终止服务/平台协商”流程，该流程不在本次实现范围。

## 3. 领域模型

### 3.1 装修需求 `service_requests`

装修需求是多位候选师傅共享的业务主对象，包含：

- 业主用户 ID。
- 工种、服务城市、服务地址、需求说明。
- 状态：`OPEN`、`COMPARING`、`WORKER_SELECTED`、`CANCELLED`。
- 最终选中的预约 ID 和报价 ID。
- 创建、更新时间与乐观锁版本。

现有预约在迁移时各自回填为一条独立装修需求，确保生产历史数据不丢失。

### 3.2 候选预约 `bookings`

每位候选师傅对应一条预约，通过 `service_request_id` 归属于装修需求。最多 3 条未结束候选预约。

预约状态扩展为：

- `PENDING`：等待师傅接单。
- `ACCEPTED`：师傅已接单，等待提出上门时间。
- `VISIT_PROPOSED`：师傅已提出上门时间，等待业主确认。
- `VISIT_SCHEDULED`：业主已确认上门时间。
- `ARRIVAL_PENDING`：师傅已标记到达，等待业主确认。
- `ON_SITE`：业主确认师傅已上门，可提交报价。
- `QUOTE_PENDING`：已有报价等待业主处理。
- `READY_TO_START`：报价已确认，等待开工。
- `REJECTED`：师傅拒单。
- `CANCELLED`：上门前由一方取消。
- `NOT_SELECTED`：业主确认其他候选师傅。

迁移同步修正现有数据库状态约束遗漏 `CANCELLED` 的问题。

预约保存拟上门时间、业主确认时间、师傅到达时间、业主确认到场时间、取消操作者、取消原因和取消时间。关键状态变化写入审计事件，便于争议追溯。

### 3.3 报价 `quotes`

沿用 V6 `quotes` 和 JSON 明细结构，在报价阶段通过新的增量迁移补充：

- `revision_number`：同一预约的报价版本号，从 1 递增。
- `supersedes_quote_id`：新版本对应的上一版本。
- `owner_decided_at`：业主处理时间。
- `rejection_reason`：业主拒绝原因。

状态为 `SUBMITTED`、`ACCEPTED`、`REJECTED`、`NOT_SELECTED`。同一预约同一时间只能有一张 `SUBMITTED` 报价；`ACCEPTED` 报价永久锁定。

报价 JSON 每一项保存完整快照：

- `templateItemId`
- `name`
- `category`：人工、辅料或主材
- `spec`
- `platformUnitPrice`
- `quantity`
- `unit`
- `subtotal`
- `remark`

响应同时返回人工费、辅料、主材和总价汇总。

## 4. 平台固定价格

后端提供稳定 ID 的工种报价目录，是价格权威来源。Flutter 不再以本地模板作为提交价格的依据。

工人读取指定预约的报价模板时，后端根据预约工种返回允许的项目、规格、单位和固定平台价。提交请求只包含模板项目 ID、规格、数量和备注。后端执行以下校验：

- 预约属于当前工人且状态为 `ON_SITE`。
- 项目属于该预约工种。
- 规格属于平台允许列表。
- 数量为正数且不超过业务上限。
- 单价、小计和总价全部由服务器计算。

服务器保存当时的平台价快照，后续平台调整价格不会改写历史报价。

## 5. 端到端流程

### 5.1 候选师傅

1. 业主创建装修需求。
2. 业主从同一工种的公开师傅目录中添加候选师傅。
3. 后端在事务中检查需求归属、工种一致、师傅资料完整、没有重复候选、活动候选不超过 3 位。
4. 每位师傅只收到自己的预约。
5. 业主可在“我的家”查看候选进度；工人看不到其他候选身份和数量。

### 5.2 上门确认与取消

1. 师傅接单后提出上门时间。
2. 业主确认后预约进入 `VISIT_SCHEDULED`；若时间不合适，业主拒绝该时间并填写原因，工人重新提出。
3. 上门时间到达后，师傅点击“我已到达”，预约进入 `ARRIVAL_PENDING`。
4. 业主点击“确认已上门”，预约进入 `ON_SITE`。
5. 在 `ON_SITE` 前，业主或师傅均可取消，必须选择或填写原因。
6. `ON_SITE` 后普通取消接口拒绝请求，后续由独立的终止服务流程处理。

### 5.3 报价、拒绝与重报

1. `ON_SITE` 的师傅读取服务器报价模板。
2. 师傅选择项目、规格和数量，填写可选备注；平台单价只读。
3. 后端校验并计算后保存新报价版本，预约进入 `QUOTE_PENDING`。
4. 业主“我的家”和消息页从服务器状态生成“待确认报价”反馈。
5. 业主拒绝时必须填写原因；报价变为 `REJECTED`，预约回到 `ON_SITE`。
6. 工人看到拒绝原因，可基于旧版本创建下一版；旧版本不可编辑或删除。

### 5.4 业主独享报价对比与最终选择

同一装修需求下有两张及以上报价时，业主可进入报价对比页：

- 对比每位师傅的版本、总价、人工费、辅料和主材。
- 按平台项目 ID 对齐同名项目，展示数量和小计差异。
- 对缺少项目、新增项目和备注作明确标记。
- 工人接口和工人端不提供任何竞争报价数据。

业主确认报价采用防误触流程：

1. 报价详情页第一次点击只打开二次确认层。
2. 二次确认层展示师傅、地址、版本、各类费用和总价，并提示其他候选将结束。
3. 业主必须勾选“我已核对项目、数量和总金额”。
4. 最终操作必须持续按住 2 秒，中途松手不提交。
5. 后端用报价状态、需求状态和乐观锁防止重复或过期确认。

确认成功由一个数据库事务完成：

- 目标报价变为 `ACCEPTED`。
- 目标预约进入 `READY_TO_START`。
- 装修需求进入 `WORKER_SELECTED` 并记录最终预约和报价。
- 其他活动候选预约变为 `NOT_SELECTED`。
- 其他待处理报价变为 `NOT_SELECTED`。
- 双端刷新后分别显示中选或未中选结果。

## 6. API 边界

具体路径可在实施时按现有命名规范微调，但权限和职责固定：

- 业主创建、查询自己的装修需求。
- 业主为需求添加最多 3 位候选师傅。
- 工人接单/拒单、提出上门时间、标记到达。
- 业主确认上门时间、拒绝时间提议、确认到场。
- 业主或工人在上门前取消自己的预约关系。
- 工人读取指定预约的固定价格模板并提交报价。
- 业主读取指定需求的报价列表和对比结果。
- 工人读取自己的报价与业主处理结果。
- 业主拒绝报价或防误触确认最终报价。

所有查询必须从 JWT 当前用户反查归属。按预约查询报价的接口不能只凭 UUID 返回数据。

## 7. Flutter 交互

### 7.1 业主端

- 找师傅流程增加“加入候选”与候选数量提示。
- “我的家”以装修需求为主卡片，展示最多 3 位候选的接单、上门和报价状态。
- 报价入口只展示服务器真实报价；没有报价时显示明确空状态，不生成模拟金额。
- 两张及以上报价显示“报价对比”。
- 报价详情提供拒绝原因输入和防误触确认流程。
- 切换到“我的家”或消息页、App 恢复前台时刷新服务端状态。

### 7.2 工人端

- 订单详情展示上门时间提议、已确认时间、到达确认和取消状态。
- 只有 `ON_SITE` 状态显示“填写报价单”。
- 报价模板来自服务器，单价不可编辑。
- 提交时显示进行中；服务器失败时保留表单并显示中文错误，不写入本地成功记录。
- 显示历史报价版本、待确认、已确认、被拒绝原因和未中选状态。

## 8. 错误与并发处理

- 第四位候选、重复候选、跨工种候选返回 409。
- 非预约参与者查询或操作返回 404，避免泄露对象是否存在。
- 状态不允许时提交报价、取消或确认返回 409，并返回中文可映射错误码。
- 重复确认使用幂等和乐观锁保护；过期页面必须刷新后重试。
- 网络失败不产生本地成功状态；双端保留重试入口。
- 报价计算使用 `BigDecimal`，统一两位小数，禁止浮点金额落库。

## 9. 实施拆分

### 阶段 A：装修需求、多候选与上门前取消

- 建立 `service_requests`、预约归属和生产数据回填。
- 接通最多 3 位候选、双方取消和业主“我的家”候选展示。
- 验证工人看不到其他候选。

### 阶段 B：上门时间与到场确认

- 接通提出时间、业主确认/拒绝、标记到达和业主确认到场。
- 验证 `ON_SITE` 前可取消，之后普通取消被拒绝。

### 阶段 C：固定平台价报价与业主比价

- 升级 V6 报价、服务器报价目录、版本与完整明细。
- 接通工人真实提交、业主“我的家”真实查看、拒绝重报、横向对比和防误触确认。
- 事务结束其他候选并验证两端反馈。

每个阶段独立执行测试、ECS 备份与部署、生产 API 验证、双模拟器 UI 验证，再进入下一阶段。

## 10. 测试与验收

后端测试至少覆盖：

- 候选数量、重复候选、工种和所有权隔离。
- 双方上门前取消与上门后禁止普通取消。
- 时间提议、确认、拒绝、到场和业主确认状态机。
- 报价模板固定价格、篡改单价无效、金额精确计算。
- 报价版本、拒绝原因、同一时间只有一张待处理报价。
- 业主独享比价、工人无法读取竞争报价。
- 确认事务同时锁定报价、选中预约、结束其他候选。
- 重复确认和并发确认只有一次成功。

Flutter 测试至少覆盖：

- 业主添加候选、最多 3 位、取消和候选状态展示。
- 工人提出时间、到达、仅 `ON_SITE` 可报价。
- 固定单价不可编辑，提交失败不产生本地成功。
- 业主真实报价列表、对比、拒绝原因、历史版本。
- 勾选确认、按住 2 秒、防重复提交。
- 中选与未中选在两端正确反馈。

生产验收使用同一业主、至少两位工人和同一装修需求完成：分别接单、上门、报价、业主对比并确认其中一位；验证另一位自动未中选、双方数据隔离、强停重启后状态仍一致。

- 实时推送服务；本阶段通过页面切换、前台恢复和手动刷新同步服务器状态。
- 完整施工阶段、验收与售后状态机，这些在报价确认后的后续阶段实现。
