# 工人首次资料完善设计

## 目标

工人首次注册或登录后，若服务器资料不完整，必须先完善真实资料，保存成功后才能进入接单首页。业主端后续读取同一份服务器资料。

## 本次范围

- 必填：全名、服务城市、工种、工龄、日薪、自我介绍。
- 手机号来自登录账户，只读，不允许在资料页修改。
- 登录成功后以 `GET /api/v1/workers/me` 为准覆盖本地旧缓存。
- 资料保存调用 `PUT /api/v1/workers/me`；请求失败时停留在资料页，不把未保存数据当成成功状态。
- `profileComplete` 为假时进入首次资料页；保存成功并回读为完整后进入工人首页。
- 本次不实现案例、图片、预约时间与报价；它们按后续小步分别完成。

## 数据与路由

Flutter `WorkerProfile` 增加 `serviceCity`、`dailyRate` 和明确的 `profileComplete`。`WorkerApp` 根据登录态与资料完整度选择登录页、首次资料页或首页。资料页同时复用于首页内的“个人资料”编辑入口，通过 `onCompleted` 区分首次完善和普通编辑。

后端继续使用现有 `GET/PUT /api/v1/workers/me`，并把自我介绍纳入资料完整度判断，使前后端规则一致。

## 校验与错误

- 全名、城市、自我介绍去除首尾空格后不得为空。
- 工龄为 0–60 的整数；日薪为 1–99999.99。
- 网络或服务端失败显示中文错误并保留输入。
- 只有服务端保存成功后才更新本地资料和路由。

## 验证

- 后端测试覆盖缺少简介时 `profileComplete=false`，补齐后为真。
- Flutter 状态测试覆盖真实 PUT 请求体和失败不落本地。
- Widget 测试覆盖不完整资料强制进入完善页、完整资料进入首页。
- 运行聚焦测试与 `flutter analyze`。
