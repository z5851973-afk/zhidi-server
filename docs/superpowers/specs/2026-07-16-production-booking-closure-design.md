# 知底三端生产预约闭环设计

## 目标

打通 Android 业主端、Android 工人端与阿里云 ECS Spring Boot/MySQL 之间的真实预约闭环，并将同步失败从静默降级改为用户可见、可重试的状态。

本轮完成标准是：工人发布完整资料后能被业主端看到；业主预约该工人后，工人端能看到真实业主信息和待接订单；工人接单后，业主端刷新为已确认；三端读取同一条 MySQL 预约记录。

## 范围

### 包含

- Spring Boot 认证、工人公开目录、预约创建、双方列表、接单和拒单。
- 预约响应中的业主姓名和联系电话。
- 工人端 JWT 持久化与冷启动会话恢复。
- 业主端和工人端的真实服务器状态、错误提示和重试。
- Flyway 生产迁移基线和 `ddl-auto=validate` 校验。
- 后端、Flutter 聚焦测试以及 Android 双端线上联调。
- ECS 上的数据库备份、jar 备份、原子替换、健康验证和失败回滚。

### 不包含

- 报价、施工项目、验收、支付、结算、评价、聊天等新的大型业务模块。
- 域名、TLS 证书和 HTTPS 改造。本轮 Android 仍直连 `http://47.109.0.191:8080`，并将明文 HTTP 作为已知生产风险。
- Docker 或 Kubernetes 迁移。

## 现网基线

- ECS：`47.109.0.191`。
- systemd 单元：`zhidi.service`。
- jar：`/opt/zhidi/zhidi-server.jar`。
- 环境文件：`/opt/zhidi/.env`。
- Spring profile：`prod`。
- MySQL：原生 MySQL 8，仅监听 `127.0.0.1:3306`。
- 后端：监听 `*:8080`，Actuator 健康检查为 `UP`。
- 当前 systemd 以 root 用户运行。本轮不改运行用户，避免将发布与权限迁移绑定在同一次变更中。

## 架构与数据源

Spring Boot/MySQL 是预约业务的唯一事实来源。Flutter 可保留非服务端演示页面的本地数据，但已经标记为远程工人或远程预约的对象不得回退为 Mock 成功。

业主端通过公开工人目录获取远程工人 ID，通过受保护的预约 API 创建记录。工人端用持久化 JWT 拉取自己的预约。工人接单后更新后端状态，业主端在进入订单页或用户主动刷新时重新读取。

## 后端设计

### 预约快照

预约在创建时保存展示所需的业主和工人快照，避免列表查询依赖跨表懒加载：

- `owner_name`：来自当前业主资料；资料未填姓名时使用“业主”。
- `owner_phone`：来自当前认证账号手机号。该字段仅在已授权的业主本人和被预约工人接口中返回，不进入公开目录。
- 现有 `worker_name`、工种、城市、地址和备注继续作为快照。

新 Flyway 迁移为已有预约增加可空字段并尽可能回填；应用层对历史空值使用兼容显示。新预约必须写入完整快照。

### 迁移管理

- `application-prod.yml` 启用 Flyway。
- `spring.jpa.hibernate.ddl-auto=validate`，Hibernate 不在生产自动创建或修改表。
- 发布前在备份数据库上或本地 MySQL 验证全部迁移。
- 发布后检查 `flyway_schema_history` 中新版本成功记录。

### API 兼容性

不改现有 URL 和状态枚举。`BookingResponse` 只新增 `ownerName` 和 `ownerPhone` 字段，现有客户端可忽略新字段。权限仍由 JWT 主体和 OWNER/WORKER 角色决定。

## Flutter 设计

### 业主端

- 远程工人预约必须等待 `POST /api/v1/bookings` 成功后才进入成功页。
- 预约失败保留当前页面，显示后端错误或网络错误，允许重试。
- “我的订单”进入时拉取远程预约，并提供明确的刷新和失败状态。
- 远程预约使用服务器 ID 作为稳定身份，合并时不生成重复本地订单。
- `PENDING`、`ACCEPTED`、`REJECTED`、`CANCELLED` 分别映射为待接单、已确认、已拒绝、已取消，不将未知状态默认成待接单。

### 工人端

- 工人认证会话使用平台安全存储持久化，并记录到期时间。
- 冷启动时验证会话，有效则进入工人首页并拉取远程预约；失效则清理会话并回到登录页。
- 后端登录、资料发布或拉单失败时不得伪装成线上成功。只有服务器认证成功才能进入“真实在线”状态。
- 远程订单显示 `ownerName`、`ownerPhone`、服务地址、工种和备注。
- 接单和拒单必须等待后端成功才修改本地状态；失败显示可重试错误。
- 本地 Mock 订单可作为明确的演示数据保留，但在已认证的生产在线视图中必须与真实订单分区或隐藏，不得混合伪装成服务器订单。

## 错误处理与可观测性

- API client 保留后端 `code`、`message`、HTTP 状态码和 trace ID。
- 页面区分未认证、超时、网络不可用、服务器拒绝和响应格式错误。
- 401 清理本地会话并要求重新登录；其他错误保留当前数据并允许重试。
- 服务端继续通过 `X-Trace-Id` 和响应 `traceId` 便于线上排查。
- 不在日志、测试输出或文档中写入 JWT、数据库密码和短信密钥。

## 测试设计

### 后端

- BookingService 测试验证预约快照来自当前认证业主和工人资料。
- BookingController 测试验证双方列表和接单响应包含正确业主信息。
- 修复隔离 Controller 测试中 DailyReportRepository 等新增依赖导致的 ApplicationContext 失败。
- MySQL/Flyway 集成测试从空库应用所有迁移，JPA `validate` 通过。

### Flutter

- API client 测试覆盖新业主字段、trace ID 和错误保留。
- 业主状态测试覆盖创建顺序、去重、状态映射和拉取失败。
- 工人状态测试覆盖会话恢复、真实订单显示、接单成功以及失败不改本地状态。
- 修复已发现的 JWT 读取次数断言和过时文案 Widget 测试。

### 真实线上闭环

使用专用测试手机号，在 Android 双端执行：

1. 工人登录并发布完整资料。
2. 业主登录，从线上目录找到该工人。
3. 业主创建带地址和备注的预约。
4. 工人刷新后看到业主姓名、电话和待接单状态。
5. 工人接单。
6. 业主刷新后看到已确认。
7. 双端强停重启，会话与订单状态仍从服务器恢复。

## 发布与回滚

1. 在本地完成后端全测试、Flutter 全测试和 Android 构建。
2. 读取线上数据库名后，使用 `mysqldump --single-transaction` 备份到只有 root 可读的时间戳文件。
3. 备份 `/opt/zhidi/zhidi-server.jar` 和 systemd 单元定义。
4. 上传新 jar 到临时文件，校验 SHA-256 后原子替换。
5. 重启 `zhidi.service`，检查 systemd、journal、Actuator 和 Flyway 历史。
6. 任一启动或健康检查失败时，恢复旧 jar 并重启。数据库迁移必须是向后兼容的增量迁移，回滚旧 jar 不需要立即回滚表结构。
7. 健康后执行认证、目录、预约、接单和双方回读冒烟测试。

## 验收条件

- 生产 Actuator 为 `UP`，systemd 在部署后稳定运行。
- Flyway 是生产表结构的唯一修改机制，Hibernate 仅执行 validate。
- 全新测试工人能被全新测试业主从 Android 业主端找到。
- 业主从 Android 业主端创建的预约在 MySQL 中为 `PENDING`。
- Android 工人端显示真实业主姓名和电话，接单后 MySQL 为 `ACCEPTED`。
- Android 业主端刷新后显示已确认。
- 双端强停重启后仍恢复同一会话和预约状态。
- 网络不可用或服务器拒绝时，页面显示失败并可重试，不生成伪成功的远程订单。
- 后端全测试、Flutter 全测试和 Android 双 flavor 构建通过。

## 已知风险

- 公网 API 使用明文 HTTP，JWT、手机号和业务数据可被网络中间人窃听或篡改。获得域名后应立即增加 HTTPS。
- systemd 以 root 运行 Java 进程，攻击面较大。后续应迁移到专用低权限用户。
- 现有工作区有大量未提交改动。实施时只编辑本规格直接涉及的文件，不清理或回退用户其他改动。
