# 当前源码基线整理设计

## 1. 目标

在不丢失任何业务改动的前提下，把当前高度未提交的工作区整理成可恢复、可验证、可继续开发的源码基线。整理完成后应明确当前源码是否能通过后端测试、Flutter 测试、静态分析和双端 Android 构建，并区分业务文件、证据产物、依赖工具和可再生成垃圾。

本次不新增业务功能，不部署服务器，不修改生产数据库，不自动提交或推送 Git。

## 2. 当前风险

开始整理时工作区约有 178 项状态变化，包括已跟踪修改、已跟踪删除和大量未跟踪文件。未跟踪文件中同时包含新增业务源码、测试、数据库迁移、文档、仓库内 Flutter SDK、构建证据和缓存，不能使用批量清理命令。

生产已验证能力仍以 `PROJECT_STATUS.md`、源码、测试、Flyway 历史和生产回读为准。工作区存在 V10 和多候选代码不代表已经完成测试或发布。

## 3. 保护边界

以下内容必须保留：

- `zhidi_app/` 和 `zhidi_server/` 内所有业务源码、配置、迁移和测试。
- `docs/`、`scripts/`、`AGENTS.md`、`PROJECT_STATUS.md`。
- `output/apks/` 与 `output/evidence/` 中现有可安装包和联调证据。
- 仓库内 `flutter/` SDK，以及 `.m2/` 依赖缓存；它们体积较大但当前构建依赖它们。
- `.idea/`、`.vscode/` 等用户 IDE 配置先保留，只通过忽略规则减少状态噪声。
- Git 当前标记删除的文件在确认引用、替代文件和测试覆盖前不得恢复或永久清除。

禁止执行 `git reset --hard`、`git checkout -- .`、`git clean` 或等价批量破坏命令。

## 4. 可清理范围

只有同时满足“可确定由工具重新生成”和“不承载用户配置或交付证据”的内容才可删除：

- `.DS_Store`。
- Flutter 工程的 `build/` 与 `.dart_tool/`。
- Spring Boot 的 `target/`。
- 项目内临时 Flutter/Codex HOME 缓存，例如根目录和应用目录中的 `.codex-flutter-home/`。
- 测试、分析或构建产生的临时日志和覆盖率目录。

清理前先生成源码归档；无法确定用途的文件只进入待确认清单，不删除。

## 5. 基线归档

归档存放在 `output/baselines/`，文件名包含时间戳。归档包含：

- `zhidi_app/`、`zhidi_server/`、`docs/`、`scripts/`。
- 根目录项目说明、环境示例和代理规则文件。
- 当前 `git status --short`、已跟踪差异补丁、未跟踪文件清单。

归档排除 `.git/`、`output/`、仓库内 Flutter SDK、Maven 缓存、Flutter 构建缓存、Java 构建目录和 IDE缓存。归档完成后生成 SHA-256，并用归档列表抽查关键业务文件。不得把真实密码、JWT、短信验证码或服务器 `.env` 内容复制进归档报告。

## 6. 忽略规则

在不隐藏业务源码的前提下补充根目录或子工程 `.gitignore`，覆盖系统垃圾、IDE缓存、构建目录、临时 HOME、Maven缓存和测试输出。`output/apks/`、`output/evidence/` 和 `output/baselines/` 是否长期纳入版本控制不在本次决定范围内，因此现有内容保留并在报告中单独列出。

添加规则后再次检查未跟踪文件，确保 `lib/`、`src/`、`test/`、迁移、配置和文档没有被误忽略。

## 7. 已删除文件审核

对 Git 标记删除的每个文件记录：

- 原路径和原职责。
- 当前是否存在替代实现。
- 现有源码是否仍引用原路径或类型。
- 对应测试是否已迁移或被删除。
- 结论：合理删除、应恢复、或待用户决定。

本次只形成审核结论；除非测试或构建明确证明必须恢复，且恢复不会覆盖用户的新实现，否则不自动恢复。

## 8. 验证顺序

为了区分原有问题和清理造成的问题，先在清理前记录一次结果，清理后再执行相同命令：

1. `git diff --check`。
2. 后端聚焦测试和全量 `mvn test`。
3. `flutter analyze`。
4. Flutter 全量测试。
5. 构建指向 `http://47.109.0.191:8080` 的 owner 与 worker debug APK。

如果命令失败，保留完整退出码和核心错误，不能为了得到绿色结果删除测试或吞掉异常。遇到失败转入系统化调试，先定位根因，再决定是否修复；本次基线整理只修复由整理行为引入的配置或忽略错误，原有业务缺陷进入后续清单。

## 9. 交付结果

整理结束应产生：

- 一份带 SHA-256 的源码基线归档。
- 更新后的安全忽略规则。
- 清理前后文件与容量变化。
- 已删除文件审核表。
- 后端、Flutter 测试、静态分析和双 APK 构建结果。
- 两个与当前源码对应、明确指向公网 ECS 的基线 APK；只有构建成功才生成。
- `docs/` 下的基线报告，列出已验证能力、失败项、未决风险和下一开发入口。
- 与实际验证结论一致的 `PROJECT_STATUS.md` 更新。

任何失败项都必须明确列出，不能把“文件存在”写成“功能完成”，也不能把旧 APK 或旧测试结果当作本次基线验证。
