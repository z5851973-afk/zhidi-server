# Android 正式上线总路线设计

## 1. 目标与范围

本路线把知底从当前“真实登录、工人资料与案例、业主预约、工人接单、业主反馈”的最小公网闭环，推进到可正式发布的 Android 业主端、工人端和运营管理后台。

Android 是当前唯一客户端完成标准。iOS、Web 用户端和桌面用户端不属于本路线验收范围。管理后台可采用浏览器访问的 Web 应用。

本路线包含九个业务与基础设施阶段，并在开始前设置源码基线保护阶段。每个阶段必须形成可独立验证、可回滚的交付物，不能把页面存在、计划存在或本地 Mock 成功当成功能完成。

## 2. 总体架构原则

- Spring Boot REST API 和 ECS 原生 MySQL 是业务事实来源。
- Flutter 本地状态只缓存服务器已确认的数据，不得替代服务器写入结果。
- 核心业务路径不得回退到 Mock、Picsum、Firestore 或静态示例数据。
- 所有写操作采用 remote-first：服务器成功后才更新页面成功状态。
- 金额由服务器使用 `BigDecimal` 计算，客户端提交项目、规格、数量等业务输入，不提交可信金额。
- 所有对象查询从 JWT 当前用户反查归属；无权访问统一返回 404，避免泄露对象存在性。
- 关键状态变化保留操作者、时间、原因和历史版本。
- Flyway 迁移只向前增加，不修改已经部署的历史迁移。
- 每次生产发布前备份数据库、jar、systemd 配置和环境配置，发布后检查 Flyway 与健康状态。
- 不在仓库、APK或文档中写入生产密码、JWT、短信验证码或云服务密钥。

## 3. 阶段 0：源码基线保护

执行 `2026-07-17-source-baseline-cleanup-design.md`：归档当前业务源码、生成校验值、清理确定可再生成的缓存、完善忽略规则、审核已删除文件，并记录后端测试、Flutter 测试、静态分析和双 APK 构建的真实结果。

阶段 0 不恢复或删除未经确认的业务文件，不自动提交、推送或部署。它的产物是后续所有阶段共同使用的可恢复起点。

## 4. 阶段 1：V10 多候选与双方取消

以 `2026-07-17-multi-worker-quote-closure-design.md` 和 `2026-07-17-service-request-candidates.md` 为事实规则：

- 建立装修需求 `service_requests`，无损回填既有预约。
- 一个需求最多三位同工种活动候选，重复、跨工种和第四位候选返回明确冲突错误。
- 业主查看需求下全部候选；工人接口只返回当前工人的预约，不暴露其他候选身份或数量。
- 业主或工人在确认上门前可取消，原因必填并记录操作者、原因和时间。
- 一位候选取消或拒单后，业主可补选其他工人。
- Flutter“找师傅”支持新建需求或加入已有需求，“我的家”按需求展示候选数量与状态。

工作区已经出现 V10 和部分服务需求代码，但在测试、Flutter、生产发布和双真机验收完成前仍视为未完成。

## 5. 阶段 2：上门时间与到场确认

预约严格按以下顺序流转：

`ACCEPTED → VISIT_PROPOSED → VISIT_SCHEDULED → ARRIVAL_PENDING → ON_SITE`

- 工人提出具体上门时间。
- 业主确认时间，或填写原因拒绝并要求重新提出。
- 工人到达后标记“我已到达”。
- 业主确认工人已上门后进入 `ON_SITE`。
- `ON_SITE` 前双方可按阶段 1 规则取消；`ON_SITE` 后普通取消返回 409。
- 重复点击、过期页面和并发操作通过状态检查、幂等与乐观锁保护。
- 双端显示服务器时间与状态，强停重启后必须恢复一致结果。

## 6. 阶段 3：服务器固定价格报价

- 后端建立带稳定项目 ID 的工种价格目录，包含名称、分类、规格、单位、固定平台价和启停状态。
- 工人只有在自己的预约为 `ON_SITE` 时可读取对应工种模板并提交报价。
- 工人只能选择项目、规格、数量和备注，不能修改平台单价。
- 后端校验工种、规格、数量上限并计算小计、人工费、辅料、主材和总价。
- 报价保存提交时的平台价快照，后续调价不修改历史。
- 报价提交成功后预约进入 `QUOTE_PENDING`；失败时工人端保留表单并显示中文错误，不生成本地成功记录。
- 业主“我的家”只展示服务器真实报价，没有报价时显示真实空状态。

## 7. 阶段 4：拒绝重报、多人比价与最终选人

- 业主拒绝报价必须填写原因，报价进入 `REJECTED`，预约返回可重报状态。
- 工人基于上一版创建递增版本；历史版本不可编辑或删除。
- 只有需求所属业主能查看候选报价和横向对比，工人永远看不到竞争报价。
- 对比按平台项目 ID 对齐，展示工人、版本、各费用分类、总价、数量差异和备注。
- 业主确认采用二次确认层、关键信息汇总、确认勾选和持续按住两秒。
- 后端在一个事务中接受目标报价、推进目标预约、记录最终选择，并把其他活动候选及待处理报价变为 `NOT_SELECTED`。
- 状态检查、幂等与乐观锁阻止重复确认或确认过期报价。

## 8. 阶段 5：施工、节点验收与整改

- 最终选人后由服务器创建施工项目，关联需求、预约、报价和双方用户。
- 工人提交带日期、文字、施工阶段和图片的日报；业主按时间线查看。
- 平台配置施工节点，工人提交节点完成申请和证据。
- 业主验收通过或填写原因要求整改。
- 整改保留原申请、驳回原因、新证据和处理历史，不覆盖旧记录。
- 所有节点通过后进入完工验收；完工后生成可查询的装修档案和保修信息。
- 已存在的日报代码必须经过权限、生产 API、图片上传、双端状态和 Android 真机验证后才能计入完成。

## 9. 阶段 6：清除核心流程 Mock 与本地假成功

逐页建立数据来源清单，覆盖登录、资料、工人目录、案例、需求、预约、上门、报价、施工、验收和消息。核心页面只能使用 Spring Boot API。

- 删除或隔离正式路由中的 Mock、Picsum 和静态业务金额。
- 移除核心订单/资料的 Firestore 双写与回退，避免双事实来源。
- 所有 `catch` 必须映射错误、记录必要诊断并保持原状态，禁止静默吞掉服务器失败。
- 断网、超时、401、403、404、409、重复提交和服务器 500 都必须有可恢复交互。
- 强停重启、换手机登录和清除本地缓存后，业务状态必须从服务器完整恢复。
- Mock 只允许留在测试夹具、独立预览或明确标记的演示构建中。

## 10. 阶段 7：真实短信、HTTPS、对象存储与生产基础设施

- 接入真实短信供应商，验证码不在生产响应或日志中明文返回。
- 配置域名、TLS 证书和 Nginx，客户端生产构建只访问 HTTPS 443。
- 案例、日报、整改和聊天图片迁移到对象存储；数据库保存受控对象键和元数据。
- 上传校验文件类型、大小、数量和归属，使用不可预测对象名。
- 建立数据库与文件备份、恢复演练、健康检查、集中日志、异常告警和容量监控。
- 建立正式 Android 应用 ID、签名密钥保护、版本号、环境区分和可重复构建流程。
- 开发、测试、生产配置分离；生产密钥只通过受控环境注入。

## 11. 阶段 8：聊天、推送与运营管理后台

### 聊天与推送

- 只有有效需求或项目参与者可以建立会话。
- 支持文字、图片、发送状态、失败重试、时间顺序、分页和未读数。
- 消息保存在服务器，换手机和重装后可恢复。
- 预约、接单、改约、到达、报价、验收、整改和售后事件生成站内通知，并通过 Android 推送提示。
- 推送只作提醒，打开 App 后必须从服务器读取权威状态。

### 管理后台

- 管理员独立认证、角色权限和操作审计。
- 管理业主、工人、工人审核、工种、固定价格、需求、预约、报价、项目、投诉和售后。
- 支持禁用账号、上下架价格项目和人工介入异常订单。
- 敏感信息默认脱敏，危险操作二次确认并记录原因。

## 12. 阶段 9：支付、结算、售后与正式发布

- 以业主已确认报价为支付金额依据，服务器生成支付订单。
- 支持阶段付款、支付回调验签与幂等、退款、取消、失败重试和对账。
- 建立平台服务费、工人待结算、可结算、已结算和异常冻结记录。
- 资金流水不可由客户端修改，不得使用浮点金额。
- 售后关联原需求、项目、报价、验收、付款和聊天证据，支持申请、平台处理、退款或关闭。
- 完成隐私政策、用户协议、账号注销、个人数据处理、权限最小化及发布前合规核对。
- 完成正式签名 APK/AAB、真机兼容测试、灰度发布、监控观察和回滚演练。

支付、结算和合规属于高风险范围，正式接入前必须按所选供应商和发布地区的当期规则重新核对，不能仅依据旧文档实现。

## 13. 统一阶段验收门槛

每个阶段只有同时满足以下条件才能标记完成：

1. 数据库迁移在空库和现有数据升级场景均通过。
2. 后端服务、控制器、权限、并发和错误路径测试通过。
3. Flutter API client、状态层和页面 Widget 测试通过。
4. `flutter analyze` 无问题。
5. owner 与 worker Android debug APK 使用公网测试环境构建成功。
6. 双真机完成“本端操作 → 对端刷新/推送可见 → 强停重启状态仍一致”。
7. 断网和服务器失败不产生本地假成功。
8. 生产发布前有可验证备份，发布后健康检查、Flyway 和关键 API 回读通过。
9. 保存 APK、截图、测试结果、生产备份路径和已知风险。
10. `PROJECT_STATUS.md` 只记录已验证事实，并保留后续缺口。

任一门槛未通过时，阶段状态保持未完成，不允许用手工改数据库、Mock 数据或旧 APK 代替验收。

## 14. 马维斯交付格式

马维斯每个阶段结束时提交一份阶段报告，至少包括：

- 本阶段目标与完成边界。
- 修改、新增和删除的文件清单。
- Flyway 版本与升级/回滚说明。
- 新增和变化的 API及权限规则。
- 后端与 Flutter 验证命令、退出码和测试数量。
- 两个 Android APK 的绝对路径、大小和 SHA-256。
- 双真机操作步骤和截图路径。
- 生产备份路径、部署时间、健康检查与回读结果。
- 未完成项、风险和下一阶段前置条件。

未经用户明确要求，不自动提交、推送、创建 PR 或部署生产。执行生产变更时必须在变更前再次确认授权。
